# Reference Board Export Download Plan

## Why Download Is Needed

Reference board exports are useful as review artifacts when operators need to share, archive, or compare selected output references without committing media binaries or moving source assets.

M34.25 makes JSON and Markdown exports visible in the dashboard as read-only panels. A later download flow should let an operator save those same review artifacts from the browser without creating server-side export files.

## Current Export UI State

Current dashboard export behavior:

- Export JSON opens a read-only formatted JSON panel.
- Export Markdown opens a read-only Markdown text panel.
- Copy may be available when browser clipboard access is available.
- No file download is implemented yet.
- No ZIP or PDF export is implemented.

## Recommended Approach

Prefer response-only download with `Content-Disposition`.

Do not write export files to runtime by default.

Use explicit download endpoints instead of adding a query parameter to existing export endpoints. This keeps read-only panel exports separate from browser attachment behavior.

## Response-Only Download

Download responses should be generated in memory from existing export helpers and returned directly to the browser.

Do not create:

- `/home/cuneyt/MoE/runtime/reference-boards/exports`
- runtime export files
- temporary ZIP files
- temporary PDF files
- copied source assets

If a runtime export archive is ever needed, plan it as a separate explicit milestone with its own storage, cleanup, and Git safety review.

## Explicit Download Endpoints

Recommended future endpoints:

```text
GET /gateway/media/reference-boards/{board_id}/download/json
GET /gateway/media/reference-boards/{board_id}/download/markdown
```

Avoid `?download=1` for the first implementation unless there is a strong compatibility reason. Explicit paths are easier to audit and document.

## Content-Type Rules

JSON download:

```text
application/json
```

Markdown download:

```text
text/markdown; charset=utf-8
```

## Content-Disposition Rules

Download endpoints should return:

```text
Content-Disposition: attachment; filename="reference-board-{board_id}-{YYYYMMDD-HHMMSS}.json"
Content-Disposition: attachment; filename="reference-board-{board_id}-{YYYYMMDD-HHMMSS}.md"
```

The filename must be generated by Gateway from sanitized values only.

## Filename Rules

Use safe deterministic names:

```text
reference-board-{board_id}-{YYYYMMDD-HHMMSS}.json
reference-board-{board_id}-{YYYYMMDD-HHMMSS}.md
```

Filename inputs:

- sanitized `board_id`
- UTC timestamp generated by server
- fixed extension selected by endpoint

Do not use board title in filenames unless a later milestone defines a strict sanitization policy.

## JSON Download Behavior

The JSON download endpoint should:

- reuse the existing JSON export helper
- return the same review pack content as the non-download JSON endpoint
- use `application/json`
- include `Content-Disposition: attachment`
- use a safe `.json` filename
- avoid absolute host paths, model files, secrets, and source asset bytes

## Markdown Download Behavior

The Markdown download endpoint should:

- reuse the existing Markdown export helper
- return the same Markdown text as the non-download Markdown endpoint
- use `text/markdown; charset=utf-8`
- include `Content-Disposition: attachment`
- use a safe `.md` filename
- avoid image embeds, SVG embeds, absolute host paths, model files, secrets, and source asset bytes

## Why Runtime Export Files Are Deferred

Runtime export files create lifecycle questions:

- where files are stored
- how long files remain
- who can delete them
- whether they should appear in output cards
- whether they need metadata sidecars
- how they are excluded from Git

M34.26 defers runtime export files so the first download implementation can remain response-only.

## Why ZIP/PDF Are Deferred

ZIP and PDF exports increase risk because they may imply asset bundling, rendering, embedded media, or larger temporary artifacts.

ZIP/PDF should remain separate future milestones with explicit security and cleanup policies.

## Safety Rules

- no arbitrary filesystem browsing
- no asset copy
- no asset move
- no asset delete
- no asset rename
- no generation trigger
- no shell execution
- no ZIP
- no PDF
- no embedded source images
- no embedded SVG
- no absolute host paths in exported content
- no model files
- no secrets
- no external upload

## What Is Not Implemented Yet

- no download endpoints
- no dashboard download buttons
- no runtime export folder
- no ZIP export
- no PDF export
- no asset copy mode
